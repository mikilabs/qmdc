# QMD-62: Findings

Technical findings for offline documentation search in the qmdc MCP server.
The lexical-quality question was settled with a working prototype (`artifacts/`)
before committing to a design.

## MCP tool surface & embedding pattern [[qmd62_finding_surface: Finding]]

How the new tools fit the existing Rust MCP server.

- category: docs
- related_to: [[#qmd62]]
- solution: add two `#[tool]` methods + param structs over a build-embedded corpus, reusing the shared envelope

### Integration points [[qmd62_surface_points: text]]

- The MCP server is Rust (`qmdc-rs/src/mcp`, rmcp `#[tool_router]`), stateless; tools are added as a `Parameters<T>` param struct (`params.rs`) + a `#[tool]` method (`tools.rs`) + a core op returning the `{success, ...}` / error envelope. Schemas are generated from the param structs — no hand-written JSON schema.
- The corpus is embedded like `core::guide` (`include_str!`, "version-locked to the binary"). For a whole tree, vendor a snapshot into the crate (e.g. `qmdc-rs/src/embedded-docs/`) via a `make docs-sync` mirror of `make guide-sync`, generated by `build.rs` (or the `include_dir` crate).
- `qmdc_search_docs` / `qmdc_get_doc` take no `path` (corpus is not the user workspace), mirroring `qmdc_get_guide`. `get_doc` must reject anything outside the embedded set, preserving the `force_root` fail-closed boundary.
- Optionally mirror as `qmdc://doc/<path>` resources, but keep the tool as the primary path — `core::guide` already notes some clients bridge only tools, not resources.

## Corpus size & "almost free" cost [[qmd62_finding_cost: Finding]]

Whether embedding the docs and building an index is cheap enough.

- category: docs
- related_to: [[#qmd62]]
- solution: embed markdown, build index on first use; cost is negligible at this corpus size

### Measurements [[qmd62_cost_numbers: text]]

- Corpus: 71 `*.qmd.md` files under `docs/` (excluding `tracking/`), ~300 KB of text → 1069 heading-level chunks (`artifacts/chunk_docs.py`, `artifacts/docs.jsonl`).
- +~300 KB to the binary (already multi-MB) is negligible; building an in-memory index over 1k chunks is sub-second and can be done lazily on first search behind a `OnceLock`.
- No new runtime dependency is strictly required for a baseline: FTS5 is already compiled into the bundled SQLite (`rusqlite` `bundled`). tantivy is an upgrade for quality, not a necessity — keep the engine behind a core op so it is swappable.

## Lexical-quality eval: plain tantivy on our docs [[qmd62_finding_eval: Finding]]

Does pure lexical search (BM25, no embeddings) rank our docs well? Prototype says yes.

- category: docs
- related_to: [[#qmd62]]
- solution: per-heading chunks + title boost + char-trigram + analyzer-consistent fuzzy + edismax query assembly; no synonyms

### Prototype [[qmd62_eval_proto: text]]

`artifacts/` holds a runnable tantivy toy crate (`cargo run`): `chunk_docs.py` chunks `docs/` by heading into `docs.jsonl` (fields `path`/`title`/`body`); `src/main.rs` indexes it in-memory (English stemming on title/body, a char-trigram field over title+body) and runs a 13-query battery in two modes — baseline (QueryParser OR, title^3) and "best" (exact-phrase boost title^4/body^2 + per-term title^3/body^1 + analyzer-consistent fuzzy d=1 + trigram overlap). Outputs captured in `results.txt` (baseline) and `results-best.txt` (final).

### Results [[qmd62_eval_results: text]]

The "best" assembly puts the correct section in the top 1–3 for essentially every meaningful query, and beats the baseline on ranking margin:

- `semantic search` → top-1 "Semantic Layer" (was a random "search" section).
- `vscode extension features` → top-1 "VS Code Extension" (was "Feature").
- `typed edges`, `find all references to an object` → phrase boost gives a decisive top-1 (e.g. 113 vs next 47).
- **`referense resolusion` (double typo): baseline = no hits → best = top-1 "Reference Resolution"**, purely via the char-trigram field + analyzer-consistent fuzzy. No synonyms.

### The one honest lexical gap [[qmd62_eval_gap: text]]

`render docs to a website` failed initially (docs say "SSG"/"mkdocs"/"static site", never "website") — a pure vocabulary mismatch that lexical search cannot fix without synonyms or embeddings, both out of scope. **Closed by editing the docs content, not the engine**: added "a documentation website you can host anywhere" to the intro of `docs/ssg/readme.qmd.md` and "Documentation Website Builder" to its namespace heading. Result: that page moved from outside the top-5 to **top-1** in best mode. This establishes the maintenance pattern — fix recall gaps in the docs, keep the engine cheat-free.

### Engine choice [[qmd62_eval_engine: text]]

tantivy and SQLite FTS5 are the same scoring family (BM25); tantivy's real edge is first-class fuzzy + better tokenization, which the prototype exercises. The dominant quality levers (heading chunking, title boost, trigram field, query assembly) are engine-independent, so the decision is reversible — wrap the engine in a core op. Recommendation: tantivy for quality; fall back to FTS5 only if the added binary size/dep is unwanted.

### Caveat [[qmd62_eval_caveat: text]]

The `docs/ssg/readme.qmd.md` content edits are already applied to the live docs. The `.qmdc-semantic` index was NOT reindexed against them — run `make semantic-index WS=./docs` before a docs-site/hints refresh.
